# Agent Workflow Instructions — Emby C# to Go Migration

> **Purpose:** Standardized workflow for all agents participating in the Emby C# to Go migration.
> **Scope:** Task claiming, completion protocol, commit conventions, merge conflict handling.
> **Applies to:** All agents working on `emby-go` migration tasks listed in `000.0-Migration-TOC.md`.

---

## 1. Before Starting Any Task

1. **Read the Master TOC** at [`.plan.d/000.0-Migration-TOC.md`](./000.0-Migration-TOC.md) to understand the overall plan structure.

2. **Read the Agent Workflow** (this document) in full before starting work.

3. **Read the relevant plan documents** for your assigned task:
   - Task-specific implementation details in [`.plan.d/001.1-Implementation-Phases.md`](./001.1-Implementation-Phases.md)
   - C# analysis in [`.plan.d/001.2-CSharp-Analysis.md`](./001.2-CSharp-Analysis.md)
   - API specification in [`.plan.d/002.0-API-Specification.md`](./002.0-API-Specification.md) (if applicable)

4. **Verify the current task status** in the Task Index in `000.0-Migration-TOC.md` before starting.

5. **Identify dependencies** — ensure all prerequisite tasks are complete or verify with another agent.

6. **Notify the coordinating agent** that you are starting task `<ID>` by posting in the shared context.

## 2. Task Claiming Protocol

When assigned a task (e.g., "Implement task 4.9: Stream pooling manager"):

### Step 1: Mark Task as In Progress

Update the task status in `000.0-Migration-TOC.md` Task Index:

```
| 4.9 | Implement stream pooling manager | 🔄 IN PROGRESS | <your-agent-id> | <date> | — | 4.7, 4.8 |
```

### Step 2: Create a Feature Branch

```bash
cd /home/mlapointe/git/Emby/emby-go
git checkout -b task/4.9-stream-pooling-manager
```

### Step 3: Implement the Task

- Follow Go coding conventions (see `CONTRIBUTING.md` if it exists)
- Write unit tests for new code
- Do not skip tests to "save time"
- If you encounter a missing dependency (e.g., a function not yet implemented), create a stub and track it

### Step 4: Verify Build and Tests

```bash
cd /home/mlapointe/git/Emby/emby-go
go build ./...
go test ./...
```

### Step 5: Update Task Status

When complete, update the Task Index in `000.0-Migration-TOC.md`:

```
| 4.9 | Implement stream pooling manager | ✅ DONE | <your-agent-id> | <date> | <date> | 4.7, 4.8 |
```

## 3. Commit Conventions

### Message Format

```
<type>(<scope>): <short description>

<long description if needed>
```

**Types:** `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `plan`

**Examples:**

```
feat(streaming): implement stream pooling manager

- Add StreamPool struct with sync.Pool
- Add Acquire/Release methods
- Add stream sharing logic for Live TV
- Add user prompt for recorded media
- Fixes #42

Closes #42
```

```
fix(handlers): resolve image resizing type mismatch

- Replace GetInt() with strconv.Atoi()
- Add missing strconv import
- Fixes #38
```

### Commit Frequency

- Commit early and often — at minimum, after each subtask
- Never commit broken code
- Each commit must pass `go build ./...` and `go test ./...`

### Branch Naming

```
task/<task-id>-<short-description>
bugfix/<bug-id>-<short-description>
refactor/<scope>-<short-description>
docs/<topic>-<short-description>
```

## 4. Merge Conflict Handling

### If a Conflict Occurs

1. **Do NOT force push** to main branch
2. **Do NOT resolve conflicts by deleting other agents' changes**
3. **Communicate** with the agent whose code is conflicting

### Conflict Resolution Protocol

1. Pull the latest changes from the main branch:
   ```bash
   git fetch origin
   git rebase origin/main
   ```

2. Identify conflicting files:
   ```
   git status
   ```

3. For each conflicting file:
   - Read the conflict markers (`<<<<<<`, `======`, `>>>>>>`)
   - Understand both changes
   - Discuss with the other agent if unclear
   - Resolve to the correct implementation (may be a combination or one version)

4. Mark as resolved:
   ```bash
   git add <conflicting-file>
   ```

5. Continue rebase:
   ```bash
   git rebase --continue
   ```

6. If conflicts are too complex, abort and request a meeting:
   ```bash
   git rebase --abort
   ```

## 5. Multi-Agent Coordination

### When Working in Parallel

1. **Claim tasks** in the Task Index before starting
2. **Don't duplicate work** — if you see a task already claimed, pick another
3. **Communicate dependencies** — if task A depends on task B, don't start A until B is done
4. **Use the shared context** to broadcast progress updates

### Coordination Checkpoints

After completing a phase (e.g., Phase 3: Data Layer), hold a coordination check:

1. Verify all tasks in the phase are marked ✅ DONE
2. Run integration tests across the phase
3. Merge all phase branches to `main` after review
4. Announce completion in shared context

## 6. Code Review Requirements

### Before Requesting Review

1. Self-review your diff: `git diff main...HEAD`
2. Ensure no debug code (`fmt.Println`, `log.Debug` with secrets)
3. Ensure all new code has tests
4. Ensure `go build ./...` passes
5. Ensure `go test ./...` passes
6. Run `go vet ./...` and `golangci-lint run` if available

### Review Checklist

- [ ] Code follows Go idioms and conventions
- [ ] No hardcoded secrets or credentials
- [ ] Error handling is proper (no ignored errors)
- [ ] No race conditions (check `go race ./...`)
- [ ] Tests cover happy path and error cases
- [ ] Documentation comments on exported functions
- [ ] No unnecessary dependencies

---

## 7. When Stuck or Blocked

### If a Task is Blocked

1. **Document the blocker** in `005.0-Risks-TODO.md`
2. **Communicate** the blocker to the coordinating agent
3. **Work on something else** if possible (check Task Index for unclaimed tasks)

### If You Find Missing Requirements

1. If a C# component is missing from the implementation plan, document it in `001.2-CSharp-Analysis.md`
2. Add new tasks to `005.0-Risks-TODO.md` with status ⬜ NOT STARTED
3. Update the Task Index in `000.0-Migration-TOC.md`

### If You Need Clarification

1. Post the question in shared context
2. If urgent, ask the user directly
3. Never guess at requirements — always verify

---

## 8. Definition of Done

A task is considered **complete** when:

- [ ] All code is implemented and compiles without errors
- [ ] All unit tests pass (`go test ./...`)
- [ ] The task is marked ✅ DONE in the Task Index
- [ ] Commit follows conventions and is pushed (if applicable)
- [ ] Relevant documentation is updated
