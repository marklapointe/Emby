# Emby C# to Go Migration — Implementation Plan Overview

> **Version:** 1.0
> **Last Updated:** 2026-04-30
> **Project:** Migrate Emby media server from C# to Go
> **Goal:** Complete, feature-equivalent Go implementation compatible with existing clients

---

## 1. Executive Summary

This document describes the migration plan for converting Emby media server from C# to Go. The goal is to create a standalone Go implementation (`emby-go`) that provides the same API compatibility and functionality as the existing C# codebase while offering improved performance, reduced memory footprint, and easier deployment.

### Key Objectives

1. **API Compatibility** — All existing Emby clients must work without modification
2. **Feature Parity** — All C# features must be available in Go
3. **Database Compatibility** — Existing SQLite database must be readable
4. **Graceful Migration** — Users can switch between C# and Go implementations
5. **Improved Performance** — Go implementation should be more efficient

### Out of Scope

- Converting the C# test suite (will write new Go tests)
- Supporting .NET-specific features (WPF, Silverlight clients)
- Maintaining backwards compatibility with pre-3.0 API versions

## 7. Database Compatibility

### SQLite Schema

The Go implementation uses the **same SQLite database** as the C# implementation.

**Key tables:**
- `TypedBaseItems` — Media items (movies, episodes, songs, etc.)
- `Users` — User accounts
- `UserDatas` — User preferences and play states
- `Items` — Item metadata
- `CollectionFolders` — Library folders

### Password Hashing Compatibility

C# uses PBKDF2 with SHA256, 100,000 iterations. The Go implementation must use identical parameters.

**See:** [001.2-CSharp-Analysis.md](./001.2-CSharp-Analysis.md) Section 3.2 for password hashing details.

## 9. Testing Strategy

### Test Types

1. **Unit Tests** — Repository, service, handler logic
2. **Integration Tests** — API endpoints with test database
3. **E2E Tests** — Full client workflows
4. **Performance Tests** — Benchmark critical paths

### Safety Rules

- Never connect to production database during testing
- Use isolated test database with schema copy
- Mock external services (FFmpeg, network)
- All tests must be deterministic
